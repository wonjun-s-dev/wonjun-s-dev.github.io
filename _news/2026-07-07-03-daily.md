## 1. `position: absolute`에서 width는 왜 최대한 줄어들려고 할까

```css
.box {
  position: absolute;
  top: 0;
  left: 0;
  /* width 지정 안 함 */
}
```

`position: absolute`에서 `width`를 따로 지정하지 않으면(`width: auto`), 기본 동작은 **shrink-to-fit**이다.

### Shrink-to-fit이란

일반 블록 요소(`position: static`)는 `width: auto`일 때 **부모 너비만큼 꽉 채우는** 게 기본 동작이다. 반대로 `position: absolute`(그리고 `float`, `inline-block`도 유사)는 **콘텐츠가 필요로 하는 만큼만** 너비를 최소화하려고 한다.

```html
<div style="position: relative; width: 500px;">
  <div style="position: absolute;">
    짧은 텍스트
  </div>
</div>
```

이 경우 부모의 500px을 다 채우지 않고, "짧은 텍스트" 콘텐츠가 차지하는 만큼만 좁게 줄어든다.

### 정확한 계산 규칙

브라우저는 다음 세 가지 값 중 하나를 기준으로 너비를 정한다.

1. **preferred width** — 줄바꿈 없이 콘텐츠를 한 줄로 쭉 늘어놨을 때의 너비
2. **preferred minimum width** — 단어 단위로 줄바꿈이 가능한 최소 너비 (가장 긴 단어의 너비)
3. **containing block에서 사용 가능한 너비** (top/left/right/bottom으로 제약된 공간)

이 중 **가능한 한 좁게** 만들되, 사용 가능한 공간보다 넘치지 않는 선에서 결정된다.

- 안에 텍스트가 짧으면 → 텍스트 길이만큼만 좁게
- 안에 긴 문단이 있으면 → 어느 정도 넓어지되, 화면 끝까지 안 늘어나고 적당히 줄바꿈되는 지점에서 멈춤

### `left`와 `right`를 동시에 지정하면 달라짐

```css
.box {
  position: absolute;
  left: 0;
  right: 0;
  /* width 지정 안 해도 자동으로 꽉 참 */
}
```

`left`, `right`를 둘 다 지정하면(`width: auto`인 상태에서) shrink-to-fit이 아니라 **그 사이 공간을 꽉 채우는 방식**으로 계산된다. `margin: auto` + `top/bottom` 동시 지정으로 수직 중앙 정렬되던 케이스와 같은 원리로, 브라우저가 "이 요소는 left부터 right까지의 여백을 다 채워야 한다"고 판단하기 때문이다.

### 정리

| 상황                                          | width 동작                                   |
| --------------------------------------------- | -------------------------------------------- |
| `width` 미지정 + `left`만 지정 (또는 `top`만) | shrink-to-fit (콘텐츠만큼 좁게)              |
| `width` 미지정 + `left`와 `right` 둘 다 지정  | 그 사이 공간을 꽉 채움                       |
| `width: 100%` 명시                            | 부모(containing block) 너비 기준으로 꽉 채움 |

→ "왜 absolute 요소가 부모 너비만큼 안 늘어나지?"라는 의문이 들면 대부분 이 shrink-to-fit 때문. 꽉 채우고 싶으면 `width: 100%`를 명시하거나 `left: 0; right: 0;`을 같이 쓰면 된다.

## 2. `position: fixed`도 동일하게 적용된다

`absolute`와 `fixed`는 둘 다 **out-of-flow positioned element**로 분류되기 때문에, width 계산 규칙이 같다. CSS 스펙에서 두 값은 containing block을 구하는 방식만 다를 뿐, **너비/높이 계산 알고리즘 자체는 동일한 그룹**으로 묶여있다.

| 상황                                                   | absolute          | fixed                |
| ------------------------------------------------------ | ----------------- | -------------------- |
| `width` 미지정 + 오프셋 하나만 (`top`만 또는 `left`만) | shrink-to-fit     | shrink-to-fit (동일) |
| `width` 미지정 + `left`와 `right` 둘 다 지정           | 꽉 채움           | 꽉 채움 (동일)       |
| `width: 100%` 명시                                     | 부모 기준 꽉 채움 | 뷰포트 기준 꽉 채움  |

차이가 나는 지점은 오직 **기준(containing block)이 뭐냐**뿐이다.

- `absolute` → 가장 가까운 positioned 조상 기준
- `fixed` → 뷰포트 기준 (조상에 `transform`/`filter`/`perspective`/`will-change: transform`이 있으면 그 조상 기준)

### 실무에서 흔히 겪는 문제 — 모달 오버레이

```css
.modal-overlay {
  position: fixed;
  top: 0;
  left: 0;
  /* width, height 안 줌 */
}
```

모달 배경을 화면 전체로 덮으려고 만들었는데 `width`/`height`를 안 주면 shrink-to-fit 때문에 콘텐츠 크기만큼만(거의 0에 가깝게) 줄어들어서 "왜 배경이 안 덮이지?" 하는 문제가 생긴다. 명시적으로 채워줘야 한다.

```css
.modal-overlay {
  position: fixed;
  top: 0;
  left: 0;
  width: 100%;
  height: 100%;
  /* 또는 */
  inset: 0; /* top: 0; right: 0; bottom: 0; left: 0; 를 한 번에 */
}
```

`inset: 0`처럼 네 방향을 다 지정하면 `width`/`height`를 안 써도 자동으로 꽉 채워지니, 모달 오버레이를 만들 때 이 방식을 자주 쓴다.

### TypeScript 프로젝트에서의 참고

React/Next.js 등에서 모달 컴포넌트를 만들 때, 오버레이 스타일을 인라인이나 styled-components로 관리한다면 `inset` 속성을 타입 걱정 없이 그대로 쓸 수 있다. 다만 오래된 브라우저 대상(구형 Safari 등)이면 `inset` 대신 `top/right/bottom/left`를 각각 명시하는 게 호환성 면에서 더 안전하다. DevOps 파이프라인에서 브라우저 호환성 테스트(BrowserStack, Playwright 등)를 CI에 넣는다면 이런 최신 CSS 속성 사용 여부도 같이 체크하면 좋다.

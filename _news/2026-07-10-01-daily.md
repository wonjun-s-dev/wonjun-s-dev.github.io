## 1. `margin: auto`는 언제 가로 정렬에 작동할까

```css
.btn {
  display: flex;
  padding: 0 10px;
  /* width 지정 안 함 */
}

.item .btn {
  margin: 0 auto 14px;
}
```

`width`가 없는데도 `.btn`이 가운데 정렬되는 걸 보고 "auto margin은 그냥 되는 거구나" 하고 넘기기 쉬운데, 사실 조건이 꽤 까다롭다.

### 작동 조건

1. **블록 레벨 박스**여야 한다 (`display: block`, `flex`, `grid`, `table` 등)
2. **일반 흐름(normal flow)** 안에 있어야 한다 (float, absolute/fixed 아님)
3. 요소의 **계산된 너비가 containing block보다 좁아야** 한다

세 번째 조건이 자주 간과된다. `width: auto`인 일반 block 요소는 부모 너비를 꽉 채우는 게 기본 동작이라, 채울 여백 자체가 없어서 `margin: auto`가 있어도 시각적으로 아무 일도 안 일어난다.

### 그런데 왜 `.btn`은 좁아졌나

`<button>`, `<input>`, `<select>` 같은 **폼 컨트롤 요소**는 일반 block 요소와 auto-width 계산 방식이 다르다. `display`를 `flex`로 바꿔서 블록 박스가 되어도, 너비는 부모를 꽉 채우지 않고 **콘텐츠(패딩 + 텍스트) 크기만큼만 shrink-to-fit**으로 잡힌다.

```html
<div class="item" style="width: 300px;">
  <button class="btn">확인</button>
</div>
```

`.item`은 300px이지만 `.btn`은 "확인"이라는 텍스트 + 좌우 패딩 20px만큼만 너비를 가진다. 그 차이만큼 `margin: 0 auto`가 좌우로 균등하게 분배되어 중앙 정렬된 것처럼 보인다.

만약 같은 스타일을 `<div class="btn">`에 줬다면 얘기가 다르다. div는 폼 컨트롤이 아니므로 `display: flex`를 줘도 block 박스 규칙을 그대로 따라 부모 너비를 꽉 채우고, `margin: auto`는 사실상 무효가 된다.

## 2. `margin: auto`가 작동하지 않는 경우

### (1) inline / inline-block 요소

```css
.link {
  display: inline-block;
  margin: 0 auto; /* 안 먹힘 */
}
```

CSS 스펙상 인라인 레벨 박스의 가로 auto margin은 **0으로 계산**된다. `<button>`의 기본 `display` 값이 `inline-block`인 브라우저도 있는데, 이 상태에서 `display`를 건드리지 않고 `margin: 0 auto`만 주면 절대 중앙 정렬되지 않는다. 이번에 `.btn`이 정렬된 건 `display: flex`로 블록화했기 때문이지, 폼 컨트롤이라서 저절로 되는 게 아니다.

### (2) width auto가 이미 부모를 꽉 채운 경우

```css
.box {
  display: block;
  margin: 0 auto; /* width 없음 → 이미 100% → 효과 없음 */
}
```

일반 div, section 같은 block 요소는 애초에 채울 여백이 없다. 이 경우 `width`를 명시적으로 줘야 `margin: auto`가 의미를 가진다.

```css
.box {
  display: block;
  width: 200px;
  margin: 0 auto; /* 이제 작동 */
}
```

### (3) float 요소

```css
.box {
  float: left;
  margin: 0 auto; /* 안 먹힘 */
}
```

float 요소도 shrink-to-fit 너비를 갖긴 하지만, auto margin으로 중앙 정렬하는 스펙 규칙 자체가 float에는 적용되지 않는다. float를 가운데 두려면 부모에 `text-align: center`를 걸거나(inline 요소처럼 취급), flex/grid로 감싸는 방식을 써야 한다.

### (4) 일반 흐름에서 세로 방향

```css
.box {
  width: 200px;
  height: 100px;
  margin: auto 0; /* 세로 중앙 정렬 안 됨 */
}
```

normal flow의 block 요소는 `margin-top`/`margin-bottom`이 `auto`여도 **0으로 계산**된다. 세로 방향 auto margin은 flex/grid 아이템이거나, absolute + `top`/`bottom` 동시 지정 상황에서만 유효하다.

### (5) table-cell

```css
.cell {
  display: table-cell;
  margin: 0 auto; /* 무시됨 */
}
```

`table-cell`은 margin 자체가 적용되지 않는 박스 타입이라 auto 여부와 무관하게 무효다.

## 3. flex/grid 아이템에서의 `margin: auto` — 오히려 제일 강력한 케이스

```css
.nav {
  display: flex;
}
.nav .logout {
  margin-left: auto; /* 남은 공간을 전부 흡수 */
}
```

flex 컨테이너의 자식에게 준 `margin: auto`는 예외적으로 매우 강하게 동작한다. 해당 방향의 **남는 공간을 전부 그 margin이 흡수**해버린다. 네비게이션 바에서 마지막 아이템만 오른쪽 끝으로 밀어내는 패턴이 대표적이다.

### 세로 + 가로 동시 중앙 정렬 트릭

```css
.wrapper {
  display: flex;
  height: 100vh;
}
.card {
  margin: auto; /* 상하좌우 모두 auto → 완전 중앙 정렬 */
}
```

normal flow에서는 세로 auto margin이 무효라고 했지만, flex 컨테이너 안에서는 상하좌우 auto margin이 전부 유효해진다. 그래서 "부모를 flex로 감싸고 자식에 `margin: auto`"가 세로+가로 동시 중앙 정렬의 국룰 트릭으로 통한다. grid도 `place-items` 대신 이 방식을 써도 동일하게 동작한다.

## 4. `position: absolute`/`fixed`에서의 `margin: auto`

지난 글에서 다룬 shrink-to-fit 내용과 이어지는 부분이다.

```css
.modal {
  position: absolute;
  top: 0;
  left: 0;
  right: 0;
  bottom: 0;
  width: 300px;
  height: 200px;
  margin: auto; /* 완전 중앙 정렬 */
}
```

`top/right/bottom/left`를 전부 0으로 주고 `width`/`height`를 명시하면, 남는 공간이 생기고 `margin: auto`가 그 공간을 상하좌우로 균등하게 분배한다. absolute 요소를 뷰포트(또는 가장 가까운 positioned 조상) 정중앙에 놓는 고전적인 방법이다. offset을 하나만 주거나(top만 지정 등) `width`를 안 주면 지난 글에서 설명한 shrink-to-fit이 우선 적용되면서 margin auto가 분배할 여백 자체가 애매해진다.

## 정리 표

| 상황                                                    | margin auto (가로)  | margin auto (세로) |
| --------------------------------------------------------- | -------------------- | -------------------- |
| block 요소, width 미지정 (꽉 참)                          | 효과 없음(이미 꽉 참) | -                     |
| block 요소, width 명시 (부모보다 좁음)                     | 작동                 | 작동 안 함 (0 계산)    |
| 폼 컨트롤(button 등) + `display: flex/block`, width 미지정 | 작동 (shrink-to-fit)  | 작동 안 함             |
| inline / inline-block                                     | 작동 안 함 (0 계산)   | 해당 없음              |
| float                                                      | 작동 안 함            | 해당 없음              |
| table-cell                                                 | 작동 안 함            | 작동 안 함             |
| flex/grid 아이템                                           | 작동 (강력)           | 작동 (강력)            |
| absolute/fixed + 4방향 offset 지정 + width/height 명시    | 작동                 | 작동                  |

→ 결론적으로 "margin auto가 왜 되는지/안 되는지"는 항상 두 가지만 확인하면 된다. **박스가 block 계열인가**, 그리고 **그 방향으로 여백이 실제로 남아있는가(flex/grid/absolute처럼 별도 규칙이 있는 게 아니라면)**.

### TypeScript 프로젝트에서의 참고

React/Next.js 컴포넌트에서 버튼류를 스타일링할 때 `<button>` 기본 `display`가 브라우저마다(특히 구형 Safari) 조금씩 다를 수 있어서, CSS reset에서 `button { display: inline-block; }` 등으로 통일해두는 경우가 많다. 이때 "분명 margin auto 줬는데 안 먹힌다"는 이슈가 생기면 reset 스타일이 `display`를 어떻게 건드렸는지부터 확인하면 된다. styled-components나 CSS Modules로 컴포넌트를 만들 때도 마찬가지라, 디자인 시스템 컴포넌트의 최상위 `display` 값을 스토리북 등으로 한 번씩 점검해두면 이런 정렬 버그를 미리 잡을 수 있다.

iPad Homepage 프로젝트에서 생각해봐야할 5가지 (1)

Created on July 10, 2026

2026   ·   CSS   JavaScript   Animation   Frontend

SOURCE ipad-homepage (fastcampus clone) · STACK vanilla HTML/CSS/JS · 다룰 기법 5가지

프레임워크 없이 애플 홈페이지를 그대로 흉내 낸 학습용 클론 저장소를 코드 리뷰하듯 훑었다. 라이브러리 하나 없이도 애플 특유의 인터랙션을 만들어낸 지점 다섯 곳을, 코드 + “왜 이렇게 짰을까”에 대한 심화 분석과 함께 정리한다.

1. 스프라이트시트 + steps(1)로 만든 무한 루프 아이콘

정보 블록(.info) 옆에 붙는 17개 아이콘은 전부 같은 @keyframes sprite-icon 하나를 공유한다. 다른 건 background-image뿐이다. 각 아이콘은 6×10 격자로 그려진 60프레임짜리 PNG이고, 키프레임은 1.67%(=1/60)마다 background-position을 한 칸씩 옮긴다.

.info .icon {
  width: 100px;
  height: 100px;
  animation: sprite-icon 3s steps(1) infinite;
}
.info .icon.icon--battery {
  background-image: url('../images/sprite_battery.png');
}

@keyframes sprite-icon {
  0.00%  { background-position: 0 0; }
  1.67%  { background-position: -100px 0; }
  3.33%  { background-position: -200px 0; }
  /* … 10%마다 줄이 바뀌며 총 60프레임 */
}

css/main.css:95–100, 1062–

왜 steps(N) 한 줄로 안 끝냈을까

프레임이 가로로 한 줄뿐이면 steps(6) 하나로 충분하다. 하지만 실제 스프라이트는 6열 10행 격자라 10프레임마다 background-position의 y값도 같이 바뀌어야 한다. CSS는 스텝 함수 안에서 “6번마다 줄바꿈” 같은 나머지 연산을 못 하기 때문에, 60개 퍼센트 지점을 전부 손으로 나열해 x·y를 동시에 지정한 것이다.

심화 — steps(1) vs steps(60), 그리고 왜 성능이 괜찮은가

animation-timing-function이 기본값(ease)이었다면 브라우저가 프레임 사이 값을 보간(interpolation)하려고 시도하면서 background-position이 미세하게 계속 움직여 어색하게 번진다. steps(1)은 각 구간 시작 시점에 딱 한 번만 값을 스냅(snap)시키므로, 60개의 정지 이미지를 순서대로 갈아 끼우는 것과 동일한 결과가 된다. steps(60, jump-none)처럼 스텝 개수를 명시하는 방법도 있지만, 이 경우 키프레임 offset과 스텝 개수가 정확히 일치해야 해서 관리 포인트가 늘어난다. steps(1) + 60개의 개별 offset 조합이 오히려 “각 프레임마다 임의의 좌표를 지정할 수 있다”는 유연성을 준다.

성능 측면에서는 background-position 애니메이션이 페인트(paint) 단계를 매 프레임 다시 태우는 비교적 무거운 속성이다. transform처럼 컴포지터 스레드에서 처리되지 않기 때문에, 아이콘 개수가 많아지면 (여기선 17개) 메인 스레드 부담이 누적된다. 실무에서 아이콘 수가 더 많아진다면 will-change: background-position을 걸어 레이어를 분리하거나, 아예 스프라이트를 <canvas> + requestAnimationFrame으로 그리는 방식으로 옮기는 게 낫다. SCSS를 쓴다면 60줄을 손으로 나열하는 대신 @for 루프로 키프레임을 생성해 실수를 줄일 수 있다.

@keyframes sprite-icon {
  @for $i from 0 through 59 {
    #{$i * (100% / 60)} {
      $col: $i % 6;
      $row: floor($i / 6);
      background-position: -#{$col * 100px} -#{$row * 100px};
    }
  }
}

2. IntersectionObserver는 클래스만 토글하고, 애니메이션은 CSS가 전담

스크롤 리빌 로직은 관찰자 하나로 모든 .info 요소를 감시한다. 뷰포트에 들어오면 하는 일은 딱 하나, .show 클래스를 붙이는 것뿐이다. 실제 움직임(translate, opacity)은 CSS transition이 담당한다.

const io = new IntersectionObserver(function (entries) {
  entries.forEach(function (entry) {
    if (!entry.isIntersecting) return;
    entry.target.classList.add('show');
  });
});
infoEls.forEach(function (el) { io.observe(el); });

.info {
  transform: translate(0, 100px);
  opacity: 0;
  transition: 1s;
}
.info.show {
  transform: translate(0, 0);
  opacity: 1;
}

js/main.js:73–84, css/main.css:81–91

역할 분리가 왜 “깔끔한” 설계인가

JS는 “봤다/안 봤다”라는 상태 판단만 하고, “어떻게 움직일지”라는 표현(presentation)은 CSS에 위임했다. 이 경계를 지키면 애니메이션 곡선이나 지속시간을 바꿀 때 JS를 건드릴 필요가 없고, 반대로 트리거 조건(threshold, rootMargin)을 바꿀 때 CSS를 건드릴 필요가 없다. scroll 이벤트 + getBoundingClientRect()로 같은 걸 구현했다면 매 스크롤마다 강제 레이아웃(forced reflow)이 발생했겠지만, IntersectionObserver는 브라우저가 컴포지터 단계에서 비동기로 교차 여부를 계산해주므로 메인 스레드 부담이 훨씬 적다.

아쉬운 점 1 — unobserve 누락

한 번 show가 붙은 뒤에도 observer.unobserve(entry.target)을 호출하지 않는다. 요소가 화면을 스쳐 지날 때마다 콜백은 계속 실행되고, 매번 classList.add가 (이미 있는 클래스에) 헛수고로 호출된다. 리빌 애니메이션은 한 번 재생되면 끝이라 동작은 같지만, 콜백 호출 자체는 줄일 수 있다.

const io = new IntersectionObserver(function (entries, observer) {
  entries.forEach(function (entry) {
    if (!entry.isIntersecting) return;
    entry.target.classList.add('show');
    observer.unobserve(entry.target); // 한 번 트리거되면 관찰 종료
  });
});

아쉬운 점 2 — threshold/rootMargin 기본값 그대로 사용

옵션 없이 생성했기 때문에 기본 threshold: 0, root: null(뷰포트)이 적용된다. 즉 요소가 1px만 걸쳐도 콜백이 발동한다. 애플 실제 사이트는 요소가 화면에 어느 정도(대략 20~30%) 들어와야 리빌이 시작되는 느낌을 준다. rootMargin: '0px 0px -15% 0px' 정도로 뷰포트 하단을 안쪽으로 당겨주면, 스크롤하다 화면 맨 아래에 살짝 걸친 요소가 바로 튀어나오는 어색함을 줄일 수 있다.

3. 배열 reverse() 하나로 만든 스태거 애니메이션

검색창을 열 때 헤더 메뉴와 자동완성 항목이 순서대로 나타나고, 닫을 때는 반대 순서로 사라진다. 비결은 transitionDelay = (index * 0.4) / length로 지연시간을 계산해두고, 열고 닫을 때마다 대상 배열을 reverse()로 뒤집는 것뿐이다.

function showSearch() {
  headerMenuEls.reverse().forEach(function (el, index) {
    el.style.transitionDelay = (index * 0.4) / headerMenuEls.length + 's';
  });
  // ...
}
function hideSearch() {
  headerMenuEls.reverse().forEach(function (el, index) {
    el.style.transitionDelay = (index * 0.4) / headerMenuEls.length + 's';
  });
  // 같은 배열을 다시 reverse() → 순서가 뒤집혀 역순으로 재생
}

js/main.js:46–70

심화 — Array.prototype.reverse()는 원본을 파괴한다

여기서 핵심은 reverse()가 새 배열을 반환하는 게 아니라 원본 배열 자체를 뒤집고 그 참조를 반환한다는 점이다(sort()도 동일). 그래서 showSearch → hideSearch → showSearch → ... 순으로 계속 토글하면 headerMenuEls는 호출될 때마다 순서가 반전되고, 짝수 번째 호출에서는 원래 순서, 홀수 번째 호출에서는 역순이 된다. 코드가 “열 때 A→B→C 순, 닫을 때 C→B→A 순”으로 항상 동작하는 것처럼 보이는 이유가 바로 이 부수효과(side effect)를 정확히 이용했기 때문이다.

다만 이건 양날의 검이다. 만약 두 함수 중 하나가 예외를 던지고 중간에 리턴하는 상황이 생기거나, 다른 코드가 headerMenuEls를 참조해서 순서를 가정하고 있다면 이 암묵적 상태 변화 때문에 버그가 생기기 쉽다. 의도가 명확하게 드러나도록 짜려면 원본을 건드리지 않는 편이 안전하다.

// 원본 순서를 유지하면서 열고 닫을 때 방향만 명시적으로 결정
function animateStagger(els, direction) {
  const ordered = direction === 'open' ? els : [...els].reverse();
  ordered.forEach(function (el, index) {
    el.style.transitionDelay = (index * 0.4) / ordered.length + 's';
  });
}

스프레드([...els])로 얕은 복사를 뜨고 reverse()를 적용하면, 원본 배열의 순서는 항상 고정되고 “이번 호출이 open인지 close인지”만으로 동작이 결정되어 호출 횟수(홀/짝)에 의존하지 않는 안전한 구조가 된다.

4. mask-image로 비디오 가장자리를 부드럽게 잘라내기

카메라 섹션의 <video> 태그에는 -webkit-mask-image로 PNG 마스크가 씌워져 있다.

.camera .stage video {
  position: absolute;
  top: 36px;
  left: 76px;
  -webkit-mask-image: url('../images/camera_video_mask.png');
}

css/main.css:676–680

clip-path가 아니라 mask-image인 이유

clip-path를 두고 굳이 마스크 이미지를 쓴 이유는 모양이 아니라 경계의 품질 때문이다. clip-path는 도형이든 SVG든 잘라낸 가장자리가 항상 칼같이 날카롭다(anti-aliasing이 약간 있어도 그라디언트 페더링은 불가능). 반면 mask-image는 마스크 이미지의 흑백(알파) 그라디언트를 그대로 투명도로 옮기기 때문에, 페더링된 부드러운 가장자리를 그대로 재현할 수 있다.

심화 — 실무에서 놓치기 쉬운 부분

이 코드에는 mask-size, mask-repeat, mask-position이 생략돼 있다. 기본값(mask-size: auto, mask-repeat: repeat)이 마스크 PNG의 실제 크기·비율과 우연히 맞아떨어져서 동작하는 것으로 보이는데, 비디오 요소 크기가 반응형으로 바뀌는 레이아웃이었다면 마스크가 타일링되거나 어긋났을 가능성이 높다. 프로덕션에서 이 패턴을 쓸 땐 아래처럼 명시하는 게 안전하다.

.camera .stage video {
  -webkit-mask-image: url('../images/camera_video_mask.png');
  -webkit-mask-size: 100% 100%;
  -webkit-mask-repeat: no-repeat;
  mask-image: url('../images/camera_video_mask.png');
  mask-size: 100% 100%;
  mask-repeat: no-repeat;
}

또한 -webkit- 접두사만 있고 표준 mask-image가 없는데, Safari는 여전히 접두사 버전이 필요하지만 Chrome/Edge(Chromium 계열)는 이미 오래전부터 비접두사 mask-image를 지원한다. 크로스 브라우저 대응이 필요하면 두 버전을 함께 선언해야 한다. 성능 면에서는 mask-image가 적용된 요소는 별도 컴포지팅 레이어로 승격되는데, <video>처럼 이미 무거운 요소에 마스크를 씌우면 스크롤 중 프레임 드랍이 발생할 수 있어 모바일에서는 마스크 크기를 필요 이상으로 키우지 않는 게 좋다.

5. 이미지 도용 방지, 세 줄이면 충분하다 (단, 한계는 명확히)

전역 img 셀렉터에 걸린 세 줄은 실무에서 자주 나오는 “이미지 우클릭 저장/드래그 방지” 요구사항의 가장 짧은 해법이다.

img {
  display: block;
  user-select: none;
  -webkit-user-drag: none;
}

css/main.css:33–37

user-select: none은 드래그로 이미지가 텍스트처럼 선택되는 것을 막고, -webkit-user-drag: none은 이미지를 끌어 데스크톱이나 다른 탭에 놓는 것을 막는다. 끼워 넣은 display: block은 보안과 무관하게, img가 인라인 요소일 때 하단에 생기는 미세한 여백(baseline gap)을 없애는 덤이다.

심화 — 이건 “방지”가 아니라 “억제”다

세 줄로 막을 수 있는 건 마우스 드래그와 더블클릭 텍스트 선택 정도의 캐주얼한 시도뿐이다. 개발자 도구 Network 탭, curl, 우클릭 컨텍스트 메뉴의 “이미지 저장”(contextmenu 이벤트를 별도로 막지 않는 한 여전히 동작), 스크린샷은 전혀 막지 못한다. 실제로 이미지 자체를 서버에서 응답으로 내려주는 이상 클라이언트 사이드에서 완벽히 막는 방법은 존재하지 않는다. 좀 더 방어를 강화하려면 아래를 함께 고려한다.

  • contextmenu 이벤트에 preventDefault()를 걸어 우클릭 메뉴 자체를 차단 (단, 접근성/사용성 저하 트레이드오프)
  • 원본 대신 워터마크가 박힌 이미지를 제공하거나, CSS background-image로 감춰 <img> 태그 저장을 우회적으로 어렵게 만듦
  • 진짜 민감한 자산이라면 서명된 URL(signed URL) + 짧은 만료시간으로 서버 단에서 접근을 통제 (클라이언트 트릭보다 훨씬 신뢰할 수 있는 방어선)

결국 CSS 세 줄은 “일반 사용자의 무심한 드래그 저장”을 막는 UX적 장치이지, 보안 대책은 아니라는 걸 팀 내에서 명확히 공유해두는 게 중요하다.

TypeScript 프로젝트에서의 참고

이 다섯 기법을 React/Next.js + TypeScript 스택으로 옮긴다면 다음 지점만 신경 쓰면 된다.

  • 스프라이트 애니메이션: @keyframes를 SCSS @for나 styled-components의 css 헬퍼로 생성하고, 아이콘 이름·프레임 수를 Record<IconName, { frames: number; sheet: string }> 같은 타입으로 관리하면 아이콘 추가 시 실수를 컴파일 타임에 잡을 수 있다.
  • IntersectionObserver: useEffect 안에서 생성하고 클린업에서 observer.disconnect()를 호출하는 커스텀 훅(useInView 류)으로 감싸면 위에서 지적한 unobserve 누락 문제가 훅 레벨에서 자동 해결된다. IntersectionObserverInit 타입을 그대로 훅 파라미터로 노출해두면 threshold/rootMargin도 호출부에서 타입 안전하게 조정 가능하다.
  • reverse() 부수효과: TS라고 막아주진 않지만, readonly T[]로 배열을 받는 함수 시그니처를 쓰면 reverse()/sort() 호출 자체가 컴파일 에러가 나서 실수로 원본을 건드리는 걸 원천 차단할 수 있다.
  • mask-image: CSS-in-JS를 쓴다면 벤더 프리픽스 자동 처리(styled-components는 stylis가 처리)를 믿고 표준 속성만 적어도 되는지 먼저 확인. 빌드 파이프라인에 Autoprefixer가 없다면 두 줄을 직접 유지해야 한다.
  • 이미지 보호: user-select/user-drag는 전역 CSS 리셋 파일에 한 번만 넣어두면 되는데, DevOps 파이프라인에서 CSP(Content-Security-Policy)나 signed URL 만료 정책 같은 서버 단 보호까지 함께 문서화해두면 “진짜 보호는 어디서 일어나는지” 팀 전체가 헷갈리지 않는다.

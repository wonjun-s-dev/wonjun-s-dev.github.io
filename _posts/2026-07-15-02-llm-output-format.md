ChatGPT가 추천한 영화 제목을, 자연어 처리(NER 등) 없이 정규식 한 줄로 클릭 가능한 링크로 바꿀 수 있을까? omdb-movie-app의 챗봇은 사용자와 대화하며 영화를 추천해주는데, 추천된 영화 제목을 클릭하면 바로 검색 결과로 연결된다. 이걸 가능하게 한 건 복잡한 자연어 처리가 아니라, "모델이 항상 정해진 포맷으로만 답하도록 프롬프트 단계에서 강제"하는 아주 단순한 전략이었다. 이 글은 그 설계 과정과, 다시 코드를 들여다보며 발견한 개선 여지까지 정직하게 기록한다.

## 1. 챗봇이 추천한 영화를 어떻게 "클릭 가능하게" 만들까?

문제를 정리하면 이렇다. LLM은 "인터스텔라 추천해요, 크리스토퍼 놀란 감독의 명작이죠" 같은 자유 형식의 자연어로 답한다. 이 응답 안에서 "인터스텔라"가 영화 제목이라는 걸 프론트엔드가 구조적으로 알아채려면 무언가 표시가 필요하다. 선택지는 두 가지였다. 응답을 받은 뒤 자연어 처리(개체명 인식 등)로 영화 제목을 추출하거나, 애초에 모델이 답할 때부터 영화 제목을 구분 가능한 포맷으로 감싸게 만드는 것. 전자는 정확도가 들쭉날쭉하고 구현 비용도 크다. 그래서 후자, 즉 "출력 포맷 자체를 프롬프트로 계약(contract)처럼 명시"하는 방향을 택했다.

## 2. 자연어 파싱 대신 출력 포맷을 프롬프트로 강제하기

시스템 프롬프트에 영화 제목은 항상 `{{한글제목//영어제목}}(출시년도)` 형식으로 감싸라는 규칙을 명시했다.

```ts
const fineTunedMessages: OpenAI.ChatCompletionMessageParam[] = [
  {
    role: 'system',
    content:
      '너는 영화 정보를 알려주는 챗봇이야! 어떤 방식으로든 답변에서의 영화 제목은 항상 ' +
      '{{한글제목//소문자영어제목}}(출시년도) 처럼 {{}} 기호로 제목을 묶어 // 기호로 한글과 영어 제목을 구분하고 ' +
      '() 기호로 영화의 출시년도를 표시해 줘! 그리고 영화의 한글 제목이 정확하지 않다면, 억지로 지어내지 말고 ' +
      '{{abc xyz}}(1234) 처럼 영어 제목만 표시해 줘.',
  },
  // ... few-shot 예시 (아래 3절 참고)
];
```

`{{ }}`로 감싸고 `//`로 한글/영어를 구분하는 이 마크업은 마크다운이나 HTML과 충돌할 일이 없는 독자적인 델리미터를 고른 것이다. "한글 제목이 확실하지 않으면 지어내지 말라"는 조건까지 넣은 이유는, 국내 미개봉작처럼 공식 한글 제목이 없는 영화에 모델이 그럴듯한 오역을 지어내는 걸 막기 위해서였다.

## 3. Few-shot 예시 설계: 몇 개, 어떤 기준으로 만들었나

규칙 설명만으로는 모델이 형식을 꾸준히 지키지 않는다. 그래서 실제 톤과 포맷을 그대로 보여주는 예시 10개를 시스템 메시지 뒤에 이어 붙였다.

```ts
{
  role: 'assistant',
  content:
    '영화 {{기생충//parasite}}(2019)은 국제적으로 큰 주목을 받은 봉준호 감독 작품이에요. ' +
    '현실적이면서도 예술적인 연출과 뛰어난 연기력으로 많은 관객들의 호응을 얻은 작품이죠.',
},
{
  role: 'assistant',
  content:
    '매트릭스 시리즈는 현재까지 총 3편으로 이루어져 있어요. {{매트릭스//the matrix}}(1999)는 ' +
    '사이버 퓨처와 현실 사이의 모순에 관한 기발한 아이디어의 인상적인 작품이에요. ' +
    '{{매트릭스 리로디드//the matrix reloaded}}(2003)는 첫 번째 영화의 이야기를 이어받아 전개되는 작품이에요.',
},
// ... 장르/시리즈별로 총 10개
```

10개를 고른 기준은 "포맷이 흔들릴 수 있는 다양한 케이스"를 커버하는 것이었다. 제목이 한 편인 경우, 시리즈물이라 여러 편의 제목이 한 답변에 몰려 있는 경우, 감독 이름 기준으로 여러 영화를 나열하는 경우 등을 섞었다. 실제로 이렇게 다양한 케이스를 보여줄수록 모델이 포맷을 벗어나는 빈도가 눈에 띄게 줄었다.

이 코드를 다시 들여다보며 발견한 개선 여지도 있다. 지금은 `assistant` 메시지만 10개 연속으로 나열되어 있고, 그 앞에 대응하는 `user` 질문이 없다. API가 role의 엄격한 교대를 요구하지 않아 동작은 하지만, few-shot의 원래 목적("이런 질문엔 이런 형식으로 답한다"는 패턴 학습)을 생각하면 각 예시 앞에 `{ role: 'user', content: '봉준호 감독 영화 추천해줘' }` 같은 질문을 붙여주는 쪽이 형식 준수율을 더 끌어올렸을 것이다.

## 4. 정규식으로 안전하게 캡처하기

모델이 포맷을 지켜서 응답하면, 프론트엔드는 그 포맷만 정규식으로 캡처해 클릭 가능한 요소로 바꾼다.

```ts
${
  typeof msg.content === 'string'
    ? msg.content.replace(
        /{{(.*?)\/\/(.*?)}}/g,
        (match, ko, en) => `<span class="movie-title" data-movie-title="${en}">${ko}</span>`,
      )
    : ''
}
```

`{{(.*?)\/\/(.*?)}}`는 `{{`와 `}}` 사이에서 `//`를 기준으로 한글/영어 제목을 각각 캡처하고, `g` 플래그로 한 응답에 여러 제목이 있어도 전부 치환한다. 결과로 화면엔 한글 제목이 보이고, `data-movie-title` 속성에 영어 제목을 심어뒀다가 클릭 이벤트에서 그 값으로 OMDb 검색을 다시 트리거하는 식으로 연결했다.

이 부분에도 다시 보니 두 가지 걸리는 점이 있었다. 하나는 `typeof msg.content === 'string'` 체크인데, OpenAI 응답의 `content`가 항상 문자열이라는 보장이 없어서(멀티모달 응답 등에서는 배열일 수 있음) 넣어둔 안전장치다. 다른 하나는 이 결과를 `innerHTML`에 그대로 꽂고 있다는 점이다. `ko`/`en` 값은 LLM 응답에서 그대로 온 문자열이라, 이론적으로는 프롬프트 인젝션으로 스크립트가 섞여 들어올 여지가 있다. 개인 학습 프로젝트라 지금은 기술 부채로 남겼지만, 실제 서비스라면 반드시 이스케이프나 `DOMPurify` 같은 sanitize 처리가 필요한 지점이다.

## 5. 한계와 리스크: 포맷을 어길 가능성은 어떻게 대비할 것인가

지금 구조의 가장 큰 약점은 "모델이 포맷을 지킬 거라는 가정" 자체가 검증되지 않는다는 점이다. few-shot으로 준수율을 끌어올렸을 뿐, 응답이 길어지거나 대화가 누적되면 모델이 형식을 놓치는 경우가 생길 수 있다. 이때는 정규식이 그냥 아무것도 매칭하지 못하고, 사용자에게는 `{{기생충//parasite}}(2019)` 같은 날것의 마크업이 그대로 노출된다 — 이 프로젝트에서는 실제로 그런 실패 케이스를 별도로 검증하거나 대비하지 않았다.

더 견고하게 만들려면 OpenAI의 `response_format`(JSON 스키마 강제)이나 function calling을 쓰는 방향이 있다. 예를 들어 "답변 텍스트"와 "언급한 영화 목록(제목, 연도)"을 아예 별도 필드로 분리해서 받으면, 정규식이 실패할 여지 자체가 없어진다.

```ts
const chatCompletion = await openai.chat.completions.create({
  messages: [...fineTunedMessages, ...messages],
  model: 'gpt-3.5-turbo',
  response_format: {
    type: 'json_schema',
    json_schema: {
      name: 'movie_reply',
      schema: {
        type: 'object',
        properties: {
          reply: { type: 'string' },
          movies: {
            type: 'array',
            items: {
              type: 'object',
              properties: {
                titleKo: { type: 'string' },
                titleEn: { type: 'string' },
                year: { type: 'number' },
              },
              required: ['titleEn', 'year'],
            },
          },
        },
        required: ['reply', 'movies'],
      },
    },
  },
});
```

이렇게 하면 커스텀 마크업과 정규식이라는 "약속에 기반한 계약"에서, 스키마 검증이라는 "구조적으로 보장된 계약"으로 옮겨갈 수 있다.

| --- | --- | --- | --- |
| 포맷 강제 | 시스템 프롬프트 규칙 + `{{한글//영어}}(연도)` 커스텀 마크업 | 정규식 한 줄로 파싱 가능, 별도 NLP 불필요 | 제목 자체에 `{{`, `//` 같은 문자가 섞이면 델리미터 충돌 가능 |
| Few-shot 예시 | `user` 없이 `assistant` 메시지 10개만 연속 나열 | 형식 준수율을 크게 끌어올림 | 질문-답변 쌍이 아니라서 패턴 학습 효과는 제한적 |
| 파싱 | `/{{(.*?)\/\/(.*?)}}/g` → `span.movie-title` 치환 | 구현이 단순하고 즉시 클릭 연동 가능 | 모델이 포맷을 어기면 검증 없이 그대로 노출됨 |
| 렌더링 | 치환 결과를 `innerHTML`에 직접 삽입 | 빠르게 기능 구현 가능 | 이스케이프 없어 XSS 가능성 존재 (기술 부채로 인지) |
| 비용 | 매 요청마다 few-shot 10개를 프롬프트에 포함 | 파인튜닝 없이 즉시 적용 가능 | 대화가 길어질수록 토큰 비용·레이턴시 누적 |

## 다음 단계로 갈 수 있는 방향

지금 구조는 "빠르게 동작하는 프로토타입"으로는 충분하지만, 실제 서비스로 키운다면 세 가지를 먼저 손볼 것 같다. `response_format`으로 출력을 스키마 강제해 파싱 실패 가능성을 구조적으로 없애는 것, few-shot 예시에 `user` 질문을 짝지어 포맷 준수율을 더 끌어올리는 것, 그리고 치환 결과를 `innerHTML`에 꽂기 전에 이스케이프 유틸을 거치는 것이다. LLM을 실제 UI 로직에 연결할 때는 "모델이 똑똑하니 알아서 잘 하겠지"가 아니라, 프론트가 안전하게 파싱할 수 있는 계약을 프롬프트와 코드 양쪽에서 명시적으로 설계해야 한다는 게 이번 작업에서 가장 크게 남은 교훈이다.
